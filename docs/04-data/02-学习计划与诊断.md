# 学习计划与诊断数据契约

- 状态：`accepted`
- 最后验证时间：2026-07-24

本文记录 P1 自适应学习基线的稳定数据边界。它补充总览
[数据设计](data-design.md)，不定义 Personal/Notebook Memory。

## 一、领域对象

### Learner Profile

最小画像只接受显式声明：年龄段、学段、声明来源和五个闭集教学偏好。`declaredByUserId` 与 `studentId` 分开保存，保留“谁作出声明”的审计事实。未知年龄按未成年人策略处理。匿名 Web P1 只允许本人或监护人声明；`school_asserted` 保留给未来经过正式身份验证的学校入口。

### Study Course

受信课程版本包含：

- 稳定 `courseSlug + version + gradeBand`；
- 6–12 个顺序连续的 Objective；
- 每个 Objective 有稳定 key、知识节点、标题、说明和最多四个前序先修；
- 3–10 道短诊断题，P1 中一个 Objective 最多由一道题覆盖；
- 诊断版本和答案键。

课程图与答案只存在于服务端目录。浏览器投影会删除 `objectiveKey` 和 `correctOptionId`。

### Study Goal

Goal 冻结 Notebook、学生、课程版本、学段、主题和用户期望。一个 Notebook 同时最多一个 `active` Goal；替换目标会归档旧 Goal，不静默改写历史诊断。

### Diagnostic

浏览器只提交 `attemptId` 与完整的 `questionId + selectedOptionId` 选择。服务端将目标投影为：

- `strength`：已诊断且正确；
- `focus`：已诊断但错误；
- `not_started`：没有诊断证据。

首个 `focus`，否则首个 `not_started`，成为可解释的 `nextObjectiveKey`。P1 只提供确定性起点；更复杂的推荐、复习调度和目标切换属于后续阶段。

## 二、物理表

| 表                     | 职责                                                       |
| ---------------------- | ---------------------------------------------------------- |
| `learner_profiles`     | 学生的最小声明、声明人、闭集教学偏好和版本                 |
| `learning_goals`       | Notebook 当前或历史学习目标及冻结课程版本                  |
| `learning_objectives`  | Goal 的有序目标图；key、知识节点和 sequence 在 Goal 内唯一 |
| `diagnostic_attempts`  | 不可变提交头、客户端 UUID、答案指纹、可信计数和时间        |
| `diagnostic_responses` | 逐题选择、服务端正确性、目标外键和判分版本                 |

`packages/db/src/schema/study.ts` 与迁移是物理事实源。字段长度同时由 Zod 与数据库 CHECK 限制，防止异常课程或输入扩张索引和审计行。

## 三、事务与幂等

建立计划分两次可重试事务：

1. 建立或恢复 Session、Notebook、公开 Artifact；
2. 同时校验 Session 学生、Conversation 所有者、Notebook 的永久所有者事实，以及当前
   未撤销、未过期的 owner Membership 后，才写画像、Goal 与 Objectives。授权行会在
   写事务中锁定，避免并发撤销后仍提交计划。

匿名身份 Cookie 只在两步都成功后写入。第一步会返回“本次是否新建 Session”的服务端事实；若第二步失败，只补偿删除本次新建、归属匹配且仍未产生任何 Goal 的 Notebook。补偿与 Goal 写入共用主体 advisory lock，因此不会删除已被并发请求成功绑定的 Notebook。恢复时若当前 Session 反查不到自己的 Goal 必须 fail-closed，不能回退到另一个 Notebook 的最新 Goal。

诊断提交在一个事务中完成：

1. 按 `attemptId` 获取 advisory lock；
2. 用受信课程从原始选择重新判分；
3. 核对 Goal、Session、学生、课程版本和持久目标图；
4. 写 Attempt 与 Responses；
5. 每道题追加一个 `assessment_graded` 可信事件；
6. 更新对应知识节点的 mastery 投影；
7. 写 `DIAGNOSE→EXPLAIN` 的 `state_transition` 事实并更新 Session 状态。

以上七步共享同一 PostgreSQL 事务；任何一步失败都会同时回滚 Attempt、Responses、事件、mastery 与 Session 状态。历史重放若发现 Attempt 已存在，会验证或补齐同一幂等状态事实。

相同 `attemptId + answers` 是安全重放；同 ID 不同答案抛出
`DiagnosticAttemptConflictError`。跨学生或归属漂移统一抛出
`StudyPlanNotFoundError`。

## 四、公开与私有边界

浏览器可以看到自己填写的学段与教学偏好、Goal、无答案诊断题面、三态目标进度、公开 Canvas 和 mastery 投影。

浏览器不能看到答案键、诊断题到内部目标的映射、数据库身份、领域事件、答案指纹、判分配置或其他学生的记录。

模型只能收到由可信画像解析的有限适配策略。它不能写画像、诊断结果、学习事件或 mastery。
