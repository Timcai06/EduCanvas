## 目标

在保持 Web-first K12 产品、Notebook/身份边界、教育可信事实和现有数据可回滚的前提下，把 Gateway、Web General、Web Teaching 三条 Turn 组合路径和两套 Tool Runtime 收敛为：一个 Gateway Control Plane、一个 Turn Application Service、一个 Context Engine、一个 AgentLoopEngine、一个 Tool Kernel，以及以教育为默认 Profile 的确定性 Domain Services。

## 范围

- 冻结并实现 transport-neutral `TurnApplicationPort`、Context/Ledger/Tool/Continuation/Trace Ports；
- additive 泛化 Turn Ledger，使通用与教学 Turn 共享 Model Run、Tool Call、Context Snapshot 审计语义；
- 生产化 Tool Kernel 和 effect ledger，适配 Local、Teaching、Node、MCP；
- 按 Gateway → Web General → Web Teaching 顺序迁移组合根；
- 实现审批 continuation、跨进程 lease/cancel、effect reconciliation 与真实脱敏 Trace；
- 将 AI SDK、MCP v1、OpenTelemetry 限制在可开关 Adapter，保留 native/旧 Adapter 回滚；
- 删除已经无生产调用者的重复 Runtime，回写架构、数据、接口与运维文档。

## 非目标

- 不重排 `apps/ + packages/` 顶层目录，不拆微服务；
- 不让普通 Turn 进入 LangGraph，不引入 Temporal/Redis/Kafka；
- 不复制 OpenClaw 全部 provider/channel/plugin，也不复制单用户宿主机信任模型；
- 不向未成年人默认开放 Shell、任意文件系统、Credential 或设备写能力；
- 不在本阶段物理删除旧教学表；停止写入与保留期验证后另立清理决策。

## 不变量

1. Gateway 是认证路由、Operation、审批、事件与投递的唯一控制面；
2. 正常 Turn 只有一个生产 `AgentLoopEngine` 组合点和一个应用服务；
3. Operation Event、Turn Ledger、Workflow Checkpoint、Worker Job、Trace、学习事实不互相冒充；
4. Tool 有效能力是 Actor、Agent、Notebook、Profile、入口、环境、安全策略的服务端交集；
5. Notebook Membership 不传播 Memory、Credential、Node 或 default grant；
6. Schema additive、入口单写迁移、旧 ID 可追溯、Adapter 可回滚；
7. 模型、客户端、MCP annotations、checkpoint 和 Trace 都不能写可信学习事实。

## 验收条件

1. 生产源码只有一个 Turn Application 实现和一个 `AgentLoopEngine` 构造位置；Gateway/Web General/Web Teaching 都调用它；
2. Web SSE 与 Gateway Event 对同一 scripted Turn 产生等价文本、工具、取消、错误和唯一终态；
3. Local/Teaching/Node/MCP Tool 都经过同一 Tool Kernel，覆盖五维权限交集、L2/L3 审批、幂等、timeout/cancel、`outcome_unknown` 和 secret 零外泄；
4. 通用与教学 Turn 都写入统一 Model Run、Tool Call、Context Snapshot 契约；学习事件继续引用可信证据，旧记录可读；
5. approval 通过后可跨进程继续；五个 kill point 无重复副作用、无双终态，N-2/N-1 与回滚 fixture 通过；
6. 跨进程 cancel/lease 在进程退出和重领后得到唯一 cancelled/failed/completed 终态；
7. `operationId + traceparent` 串联入口、Gateway、Turn、Model、Tool、Worker；默认遥测和队列不含正文、Prompt、判分键、Token、Secret、Credential、对象 key；
8. 共享 Notebook 中各 Actor 仍使用各自 Personal Agent，私人能力跨 Actor 全部 fail closed；
9. AI SDK/native Provider parity、MCP 生命周期与恶意输出、OTel exporter degraded/关闭路径均可验证且可回滚；
10. `make check`、`make integration`、生产 build、E2E、PTY smoke、Web/TUI 同 Notebook/handoff/审批/cancel 全绿；canonical 文档与实际代码一致。

## PR 切分原则

- 每个 PR 只完成一个可独立回滚的纵切；Schema、仓储、Runtime、调用方和测试尽量同 PR 闭环；
- 依赖顺序为 Contracts → Data → Kernel/Context → Gateway → General → Teaching → Continuation/Trace → Adapter/cleanup；
- 任何入口切换前先有 parity test，切换后保留上一 Adapter 至下一里程碑验证；
- 不使用长期 dual-write；若一次切换无法保证单写，停止并缩小纵切。

## 风险与回退

- **教学行为回归**：先冻结 Teaching golden transcript/可信事件，再迁 hooks；失败立即切回旧 Teaching Adapter；
- **账本双写/丢写**：入口级单写 feature flag，迁移只 additive，旧读路径保留；
- **副作用重复**：所有 write Tool 先建 effect key/ledger；不确定结果保持 `outcome_unknown`；
- **跨进程竞态**：PostgreSQL row/advisory lock、lease generation 与唯一约束；真进程 kill 测试；
- **框架泄漏**：AI SDK/MCP/OTel/LangGraph 类型只在 Adapter；依赖边界测试阻止反向导入；
- **范围过大**：按纵切独立 PR；任何阶段必须在上一阶段全绿并可回滚后开始。
