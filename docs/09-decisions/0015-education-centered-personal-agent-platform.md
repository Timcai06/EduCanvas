# ADR-0015：以教育能力为核心的通用个人 Agent 平台

- 状态：`accepted`
- 日期：2026-07-19
- 决策人：项目负责人

## 背景

EduCanvas 先后被描述为 K12 教学助手、通用全模态工作空间和带 K12 Profile 的 Agent 平台。三个表述分别抓住了竞赛场景、通用能力和运行时复用，但没有回答产品第一身份是什么，导致教学状态机、Notebook 体验和通用 Agent 能力反复争夺架构主导权。

对 OpenClaw 的研究表明，值得吸收的是“Assistant 是产品、Gateway 是控制平面、多种客户端和渠道共享同一个 Agent”的产品结构；对 Claude Code 的研究表明，一个统一循环、上下文引擎、工具权限和压缩/恢复机制可以支撑专业 Agent。EduCanvas 仍需要自己的教育可信边界，而不是复制任一产品。

## 决定

1. EduCanvas 的第一身份是**以教育能力为核心的通用个人 Agent 平台**。
2. “个人 Agent”是用户长期拥有、可持续积累上下文并能通过工具完成任务的主体；Web 页面、TUI 和消息渠道都只是它的交互表面。
3. 教育不是独立 Runtime，也不是可删除的演示插件。它是平台核心且最完整的能力域，包括年龄适配、资料学习、讲解与追问、练习与 Artifact、可信判分、学习证据和结构化课程。
4. Agent 的基础能力保持通用：文本与多模态理解、Notebook、Sources、记忆、工具、Artifact、持久任务和多渠道交互不得依赖课程状态。
5. 默认对话是具备教育能力的通用个人 Agent，不假定每位用户都在上课，也不把普通问题强制转成课程。K12 是首个正式的结构化 Profile 和当前竞赛交付面；只有需要诊断、练习、测评、掌握度或可信学习证据时，才显式进入课程 Workflow。
6. 产品不以“复制 OpenClaw”或“复刻 Gemini/NotebookLM”为目标。OpenClaw 提供 Gateway-first 参考，Claude Code 提供 Runtime 参考，EduCanvas 的差异是教育能力、Notebook、Artifact 和可信学习证据的组合。
7. 一个自然人平台用户拥有自己的个人 Agent。Agent 的私人记忆、默认 Notebook、Credential、设备配对和默认工具授权不与家庭、班级或机构成员共享。
8. 家庭、班级和协作关系通过**共享 Notebook、共享 Source/Artifact 和显式角色授权**表达，不通过“多人共用一个 Agent 身份”表达。每次操作始终保留真实 Actor、所属个人 Agent 和目标 Notebook。
9. 教师、家长或管理员可以获得明确范围、可撤销且可审计的查看或管理权限，但不能冒充学生 Agent；学生的学习证据仍按学生主体隔离。

## 后果

- 产品北极星从“一个 Web 教学页面”改为“一个可从多个表面访问的长期个人 Agent”；
- Web 仍是教育体验最完整的客户端，但不是唯一入口；
- K12 状态、掌握度和课程不进入通用消息、Gateway 或 Agent Loop 契约；
- “教育是核心”表达能力、安全与可信边界，不等于“教学模式始终开启”；
- 共享 Notebook 可以承载班级资料、家庭资源或协作产物，但不会自动暴露成员的私人对话、记忆、凭据或设备能力；
- 群聊或共享渠道线程中的每条消息仍映射到真实发送者；共享的是上下文范围，不是 Agent 身份；
- 任何新能力必须同时回答通用 Agent 价值和教育场景中的安全/可信边界；
- 竞赛简报只定义首个教育交付面，不能反向缩窄整个平台。

## 验证方式

- 产品、架构、计划和 README 使用同一产品身份；
- 一个不加载课程状态的个人 Agent 可以运行；
- K12 Profile 在同一 Runtime 上增加教育能力，而不是创建第二套循环；
- Web、TUI 或渠道发起的消息都能路由到同一用户 Agent 与 Notebook。
- 两个成员进入同一共享 Notebook 时，各自私人记忆、Credential 和设备授权保持不可见；
- 教师/家长的委托访问记录 Actor、授权来源、作用域和撤销状态，不生成冒充学生的操作记录。
