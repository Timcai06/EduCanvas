# 开发工作流

- 状态：`accepted`

[团队协作指南](team-guide.md)是新成员的最短指南，本文件说明团队内部怎样组织任务。

## 分支命名

每项工作从 `main` 创建独立分支，统一使用 `类型/YYYYMMDD-简短任务名`，例如 `feat/20260713-lesson-canvas`。日期是创建分支当天的中国时区日期，用于把并行任务按开始时间定位；commit 和 PR 标题继续使用 `类型: 做了什么`，不加入日期。完整示例和命令以[团队协作指南](team-guide.md)为准。

## 一个功能的生命周期

```text
Issue说明问题
→ 明确负责人和验收标准
→ 创建功能分支
→ 编码和本地验证
→ 更新文档
→ 创建Pull Request
→ Review和自动检查
→ Squash合并
→ 验证共享环境
```

## Issue至少写清楚

- 背景；
- 目标；
- 不做什么；
- 验收标准；
- 负责人；
- 相关文档；
- 依赖和风险。

## PR大小

优先小PR。一个PR完成一个可解释目标。数据库、接口、前端和文档如果属于同一功能可以一起提交，但不要顺手混入无关重构。

## Review重点

- 是否符合需求；
- 是否引入安全或隐私问题；
- 是否存在并发、幂等和失败恢复问题；
- 是否影响API、数据或Artifact协议；
- 测试是否覆盖关键风险；
- 文档是否同步。
