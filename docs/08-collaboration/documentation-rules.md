# 文档维护规则

- 状态：`accepted`

## 谁修改代码，谁更新文档

文档不是项目负责人最后补写的报告。实现行为发生变化时，相关PR必须同时更新文档。

## 放在哪里

- 产品行为：`01-product`；
- 跨模块架构：`02-architecture`；
- AI、RAG和模型：`03-ai`；
- 表、事件和数据治理：`04-data`；
- 具体工程规范：`05-engineering`；
- 测试、安全和评测：`06-quality`；
- 部署与监控：`07-operations`；
- 团队协作：`08-collaboration`；
- 重大决定：`09-decisions`；
- 跨阶段路线图和长期里程碑：`10-planning`；
- 正在执行的阶段计划、验收证据和收尾记录：`plan`。

## 路线图与阶段计划的边界

- `10-planning/roadmap.md`回答“项目分几阶段、每阶段长期要交付什么”，是稳定的规划入口；
- `plan/active/`回答“当前阶段具体做什么、如何验证、由谁收尾”，是短期执行工作区；
- 执行计划不得复制完整架构说明，应链接对应的canonical文档；
- 计划不是实现事实。计划中的“将实现”“待验证”不能被其他文档引用为已完成能力；
- 当计划产生新的长期技术约束或改变已接受决策时，必须新增或更新ADR，不能只改计划。

## 阶段计划生命周期

1. **创建**：从`docs/plan/_template.md`创建计划，放入`docs/plan/active/`，填写负责人、范围、非目标、验收条件和预期回写文档；
2. **执行**：任务状态和验证结果只在该计划维护，稳定设计结论同步到对应canonical文档；
3. **收尾**：逐项记录命令、测试、截图、PR或部署等证据，不得只写“已完成”；
4. **事实回写**：把已落地的产品行为、架构、接口、数据和运维事实回写到`01-product`至`07-operations`；重大取舍写入`09-decisions`；阶段进度需要时更新`10-planning`；
5. **归档重构**：删除失效的候选方案和过程性清单，保留目标、实际范围、关键偏差、证据及canonical文档链接，然后移入`docs/plan/completed/`；
6. **目录整理**：每完成一个阶段，都应清理`active/`中的失效计划、合并重复计划，并更新`docs/plan/README.md`索引，避免计划目录成为第二套知识库。

计划被取消时，将状态改为`cancelled`并归档，说明取消原因和替代计划；不要直接删除仍被PR、ADR或复盘引用的计划。

## 不要做什么

- 不建立`最终版2.md`、`最新版.md`；
- 不复制同一结论到多个文件；
- 不把聊天记录原样贴入文档；
- 不在没有承接结论和历史记录的情况下删除ADR；过时ADR应先把仍有效的约束写入当前ADR，把关键演进压缩进`09-decisions/decision-history.md`，再删除旧文件；
- 不写无法验证的完成状态；
- 不在文档中放真实密钥或学生数据；
- 不把`plan/`当作长期架构、接口或产品事实源；
- 不在`active/`长期堆积已经结束、暂停或被替代的计划。

## 文档头部

重要文档至少包含：状态、负责人、最后验证时间。暂时未知的负责人可以写`待认领`。

阶段计划还必须包含：目标、范围、非目标、验收条件、验证证据、预期回写文档和收尾检查表。
