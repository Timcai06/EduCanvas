## 二、对 Hybrid Ports 最强的反方论证

最强反方不是“框架更流行”，而是：**直接以 OpenClaw 或 LangGraph/AI SDK 组合为运行基座，可以立即获得成熟的 Gateway、渠道、session、工具、流式模型、审批、checkpoint 和生态；EduCanvas 只需增加 K12 Profile 与 UI，便能减少自研基础设施、跨入口漂移和长期维护。**

这套论证有真实优势：

- OpenClaw 已把多渠道、Gateway、session、工具和长期个人 Agent 做成完整产品闭环；
- LangGraph 已提供 checkpoint、interrupt、thread history 与故障恢复抽象；
- AI SDK 和 MCP 能减少 Provider 与外部工具协议的重复适配；
- 对小团队而言，复用社区修复通常比独立维护更快。

如果 EduCanvas 只是单用户通用助理，或可以接受框架 Session/Thread 成为产品事实，这很可能是更好的方案。

## 三、为什么当前证据仍不支持 framework-first

EduCanvas 的硬约束不是“能对话和调工具”，而是同时满足多 Actor Notebook、未成年人隐私、确定性教学事实、服务端授权、唯一 Operation 终态、可回滚数据迁移和 Web-first 产品。把框架放到最内层并不会删除这些工作，只会把它们变成框架外的补偿层：

| 候选基座         | 能直接减少                       | 仍必须由 EduCanvas 实现                                                   | 新增成本/风险                                                |
| ---------------- | -------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------ |
| OpenClaw-first   | Gateway、渠道、session、工具生态 | Actor/Agent/Membership、Notebook、K12 安全、学习事实、PostgreSQL 业务账本 | 映射或替换其信任/session/workspace 语义；跟随大体量上游演进  |
| LangGraph-first  | graph/checkpoint/interrupt       | Gateway、Operation、授权、effect ledger、消息/审计、教育事实              | 第二 checkpoint schema、图版本迁移、节点重跑、副作用仍需幂等 |
| AI SDK/MCP-first | Provider 与外部 Tool 适配        | Loop 单终态、Tool Policy、审批、取消、业务审计                            | 原始错误/工具执行语义需再次收敛；协议元数据不能当权限        |

五类仓库 fixture 给出了直接证据：AI SDK 达成 Provider parity，但只覆盖一个 Adapter；LangGraph 与原生 continuation 都需要外部 effect ledger，且 LangGraph增加两项依赖和四张 checkpoint 表；Tool Kernel 的核心复杂度是 EduCanvas 权限与副作用语义，MCP 不能消除；Privacy 与 Trace 同样依赖产品自己的 Actor 和数据分类。

## 四、30% 门槛判定

“降低 30%”必须是代表性纵切的可复现结果，不接受 star 数、宣传功能表或只比较 happy-path LOC。计算范围固定为：

```text
总成本 = 实现与迁移 + 契约/安全/故障测试 + 数据升级/回滚 + 生产运维
```

并必须同时通过：身份与 Notebook 单源、Operation 单终态、学习事实单源、五个 kill point、N-2/N-1、跨 Actor 隐私、secret 零外泄、可撤回适配器。

当前结论：**没有候选证明总成本下降 30%**。

- AI SDK 只在 Provider slice 有复用价值，不可能单独减少整套 Runtime 总成本 30%；
- LangGraph 对照未减少 effect/approval/权限工作，反而增加 checkpoint 运维面；
- OpenClaw 没有在 EduCanvas 多 Actor K12 约束下完成同题纵切，不能把其现成功能数量计为净节省；
- 多框架组合仍需要 EduCanvas Ports 来阻止第二事实源，因此支持 Hybrid Ports，而非推翻它。

这不是永久拒绝。若未来有候选在同一 fixture 上达到 `<=70%` 的基线总成本，并保留一条完整回滚路径，必须重开 ADR；有界 Workflow 若分支/并行/长等待复杂度达到相同门槛，也可单独采用 LangGraph Adapter。

## 五、决策

- `build/retain`：产品信任内核、Gateway authority、Notebook/Context、Tool Kernel、业务账本、教育 Domain Services；
- `adopt`：graphile-worker；OpenTelemetry 稳定 API/SDK（生产仍需单独接入）；
- `adapt`：AI SDK Provider、MCP v1 External Tool、OpenClaw/Codex/Claude 的产品模式；
- `defer`：LangGraph 有界 Workflow、Temporal、MCP v2 Tasks、A2A、框架 Memory/Session；
- `reject`：framework-first 重写、每个 Turn 图化、框架 Session/Checkpoint 成为 Notebook/Operation/学习事实源。
