# 系统级长期记忆研究

- 状态：`draft`
- 负责人：项目负责人
- 最后验证时间：2026-07-24
- 适用提交：`501e3cb`

本文研究 EduCanvas 如何从“单次对话上下文”演进为可解释、可纠正、可删除的三层长期记忆。
它用于形成后续 ADR 与实现计划，不代表数据库结构或产品契约已经生效。

## 一、研究问题

EduCanvas 已把 Notebook 定义为资料、目标、对话和学习产物的共同边界，但一个长期使用的
个人 Agent 还需要跨 Notebook 记住用户明确授权保存的偏好与稳定背景。同时，Conversation
又需要短期工作记忆来承载当前推理、工具结果和续跑状态。

关键问题不是“怎样存更多聊天记录”，而是：

1. 什么信息可以跨 Conversation 或跨 Notebook 使用；
2. 谁能写入、查看、更正、删除和共享这些信息；
3. 模型生成的候选记忆如何避免覆盖教学与权限事实；
4. 每次回答怎样证明用了哪些记忆、为什么使用以及使用的版本；
5. 对未成年人如何让错误记忆和隐私泄漏的代价低于“什么也没记住”。

## 二、当前仓库事实

- PostgreSQL 是身份、Notebook、Conversation、Operation 与可信学习事实的唯一事实源；
- 学习者画像、Goal、Objective、诊断、判分和掌握度已经有确定性契约与服务端写入边界；
- `Context Engine` 负责上下文编译，模型和客户端不能绕过授权与预算直接拼接事实；
- Notebook 可以被角色共享，而 Personal Agent 属于自然人；
- Conversation compaction、Notebook 摘要与 Personal Memory 尚未形成统一产品契约；
- 路线图已明确系统级、Notebook 级、会话级三层 Memory，但尚未接受具体存储方案。

因此，Memory 应作为 Context Engine 的一个有治理输入，而不是新的事实源、隐藏的系统提示词
或框架内部 checkpoint。

## 三、外部研究结论

### 3.1 MemGPT：分层上下文，不等于让模型管理事实

[MemGPT](https://arxiv.org/abs/2310.08560) 用虚拟内存类比，把有限的主上下文与外部上下文
分层，并让 Agent 在层级之间搬运信息。EduCanvas 应学习“有限上下文、分层存储、按需调入”，
但不能让模型拥有身份、权限、判分或掌握度事实的最终写权。

### 3.2 Generative Agents：检索与反思适合候选记忆

[Generative Agents](https://arxiv.org/abs/2304.03442) 使用记忆流，并以相关性、近期性和重要性
检索，再通过反思形成更高层结论。这适合组织情节记忆和候选洞察，但其目标是生成可信的模拟
行为，不是 K12 教学治理。其“重要性”和“反思”不能直接成为 EduCanvas 的权威事实。

### 3.3 Reflexion：从情节反馈提炼受控策略

[Reflexion](https://arxiv.org/abs/2303.11366) 将语言反馈保存在 episodic memory buffer 中，
用历史反馈改善后续尝试。将其中经评测或人工确认的教学策略、工具失败经验晋升为程序性候选，
是 EduCanvas 的设计推论而非论文原有分类；模型对学生人格、能力或心理状态的反思不能成为
学生画像。

### 3.4 Mem0 与 A-MEM：提取、整合和关联都应经过状态机

[Mem0](https://arxiv.org/abs/2504.19413) 强调从对话中提取、整合和检索记忆，
[A-MEM](https://arxiv.org/abs/2502.12110) 借鉴卡片笔记，以原子笔记、动态链接和演化组织
长期记忆。两者支持“原子记忆 + 关联 + 更新”的方向，但模型提取结果必须先进入候选状态，
不能自动改写已确认信息。图关系和向量数据库也不是第一阶段必需品。

### 3.5 LongMemEval：记忆质量不等于命中率

[LongMemEval](https://arxiv.org/abs/2410.10813) 将长期记忆能力拆为信息提取、多会话推理、
时间推理、知识更新和拒答。EduCanvas 还必须增加跨作用域泄漏、陈旧覆盖、更正、删除、敏感
信息和错误归因测试。对 K12，错误调用记忆通常比没有调用记忆更危险。

### 3.6 Gemini 与 NotebookLM：个人记忆和工作空间记忆是两种产品

Gemini 的[过去聊天个性化说明](https://support.google.com/gemini/answer/16598469?co=GENIE.Platform%3DDesktop&hl=en)
将跨聊天个性化设计为可开启、可询问来源和可纠正的个人能力，并限制未成年人、受监督账号
和学校账号使用。其“删除”通常是删除承载信息的聊天，生效可能延迟，连接应用还需分别断开，
并不是严格的逐条记忆即时删除。这说明面向 K12 时不能默认从聊天建立跨 Notebook 个人记忆，
也不能拿消费级产品的删除界面替代完整数据删除闭包。

Gemini 的[学习 Notebook 说明](https://support.google.com/gemini/answer/16972047?co=GENIE.Platform%3DDesktop&hl=en)
则把资料、指令、讨论、诊断、课程和进度放在一个专门空间内。它更接近 EduCanvas 的
Notebook Memory：边界清晰、来源可见、为当前学习任务服务。

## 四、三层作用域

### 4.1 Personal Memory：整个个人 Agent 的长期记忆

所有者是自然人，只服务于该人的 Personal Agent，不因 Notebook 分享而被其他成员继承。

适合保存：

- 用户明确要求“记住”的称呼、语言、交互和无障碍偏好；
- 可撤销的跨 Notebook 工作习惯；
- 已确认且仍有效的个人目标索引。

Learner Profile 已定义的讲解顺序、回答深度、引导程度、模态与反馈风格五维教学偏好不在此
重复存值；Personal Memory 只能保存 `authority_ref + version` 索引。新增长期偏好必须使用
不同语义键，并先明确它为何不属于 Learner Profile。

默认不保存：

- 从普通聊天推断出的年龄、性格、能力、心理状态或家庭情况；
- 健康、性、宗教、政治、精确位置、学校身份、生物特征；
- Credential、权限、审批决定和设备访问能力；
- Notebook 私有来源、共享成员信息和未授权教材内容。

未成年、匿名、受监督和学校账号是否能使用 Personal Memory 必须由 M0 授权矩阵决定；在
矩阵被接受前全部关闭。即使未来允许，也不能仅凭一句“记住”绕过年龄、身份和数据控制者
规则，更不能启用聊天自动抽取。

### 4.2 Notebook Memory：一个学习工作空间的集体记忆

作用域是 Notebook，可按 Notebook 角色共享。它记录当前学习项目的连续性，而不是复制完整
聊天记录。

适合保存：

- 来源目录、来源摘要、引用与版本；
- Goal、Objective 和学习路径的可读索引；
- Notebook 内已确认的术语和公共决策；
- Artifact、练习、实验和阶段成果的索引；
- 有来源支撑的 Notebook 摘要与待办。

Notebook 分享只传播 Notebook Memory，不传播任何成员的 Personal Memory。取消分享、
删除来源或成员权限变化后，Context Engine 必须立即停止调入不再授权的信息。

学生个体误区、教师反馈、学习表现和掌握度属于权威学习事实或成员私有 audience，禁止默认
提升为 Notebook-wide Memory。每条 Notebook Memory 都要声明 owner 与 audience；共享
Notebook 只代表可引用该作用域，不代表所有角色能看到其中所有记录。

### 4.3 Conversation Working Memory：当前任务的工作记忆

作用域是 Conversation，生命周期短于前两层：

- 最近消息和受预算约束的摘要；
- 当前工具结果、Operation 状态和 continuation；
- 本轮计划、未完成动作和临时引用；
- compaction 后可恢复的工作快照。

工作记忆可以被压缩、过期或归档。它不是跨 Notebook 的长期用户画像，也不能替代 Operation
账本和框架 checkpoint。

Conversation 默认继承所属 Notebook 的 ACL，但消息、工具结果与 Artifact 还要受其自身
可见性约束；私人 Conversation 不因 Notebook 分享自动公开。移动或复制 Conversation 必须
重新鉴权并重建 Context Snapshot，不能保留原空间的隐式访问能力。

三层之间没有隐式提升：

- Conversation 摘要只有经用户选择或受控规则确认后才能成为 Notebook Memory；
- Notebook 信息只有其自然人所有者重新授权后才能成为 Personal Memory；
- copy、move、share、revoke 都创建可审计的新关系，不通过改 `scope_id` 继承旧权限；
- Conversation、Notebook 或成员权限撤销后，所有上层候选必须重新验证来源授权。

## 五、Memory 与权威事实的分界

以下数据即使可被自然语言描述，也必须继续由其专用契约和服务写入，不能成为 Memory 的
可覆盖字段：

| 权威数据                             | 所有者或写入边界             |
| ------------------------------------ | ---------------------------- |
| 身份、角色、权限、Credential         | Gateway 与身份/授权仓储      |
| 年龄段、年级、声明来源及五维教学偏好 | Learner Profile 契约         |
| Goal、Objective、Session             | Study Plan 服务              |
| 诊断答案、判分、掌握度               | 确定性 Teaching Core/Runtime |
| Approval、Operation、Effect 账本     | Gateway/Tool Kernel          |
| 来源原文、Representation、Chunk      | Source 摄取管线              |

冲突时，权威事实总是胜过 Memory。Memory 只能引用事实标识和版本，不能复制后成为第二份真相。

## 六、记忆类型与正交治理维度

每条记忆同时具有作用域、类型和多个正交治理维度，不能压成一个“可信度高低”。

### 6.1 类型

- `semantic`：已确认的偏好、术语和稳定背景；
- `episodic`：发生过的学习事件、讨论和产物索引；
- `procedural`：经确认或评测的讲解、工具与学习流程经验。

### 6.2 治理维度

- `provenance`：user、authority、source、model、reflection，以及对应主体/来源 ID；
- `activation`：pending、active、superseded、revoked、expired；
- `sensitivity`：public、personal、sensitive、prohibited；
- `freshness`：事件时间、确认时间、有效期和被替代版本；
- `authority_ref`：可选的权威注册表 ID 与版本，只保存引用，运行时回查。

用户声明不天然胜过受信来源，来源可追溯也不代表内容真实或安全。共享成员上传的恶意资料、
摘要中的指令注入和错误教材都可能是 `source` provenance。冲突裁决必须由作用域、语义键、
权威注册表和显式产品规则完成，不能按 provenance 排全局高低。

model 和 reflection provenance 默认不能进入 Personal Memory 的生产检索结果；只有经过用户
确认、确定性规则或评测门后才能激活。来源内容永远作为不受信数据传入，不获得系统指令权。

## 七、写入生命周期

建议状态机：

```text
pending -> active -> superseded
   |          |
   +--------> revoked
   |          |
   +--------> expired
```

允许的写入入口：

- 当前回合用户明确发出“记住/忘记/更正”意图；
- 确定性服务将权威事实投影为只读检索索引；
- Notebook 来源摄取产生带引用的候选摘要；
- 模型提取产生 `pending model_candidate`；
- 评测或人工确认的反思进入程序性候选。

任何写入都要保存主体、作用域、来源、写入者、治理维度、版本和时间。模型不能直接把候选
升级为 `active user`，也不能静默合并相互冲突的记忆。

## 八、检索与 Context Snapshot

检索顺序必须先治理、后排序：

1. 验证当前主体、Notebook 角色、Conversation ACL 和记录自身可见性；
2. 按作用域、activation、freshness 和 sensitivity 过滤；
3. 处理更正、撤销、supersede 与来源删除；
4. 对 authority 引用运行时回查，对 source 内容做指令隔离和来源风险处理；
5. 在每层独立预算内按相关性、近期性和任务价值排序；
6. 由 Context Compiler 合并权威事实、Notebook 来源、Memory 和工作记忆；
7. 保存 Context Snapshot，记录使用的记忆 ID、版本、作用域、选择原因和预算。

建议预算顺序不是固定 token 百分比，而是按任务模板分段：

```text
系统/安全边界
-> 权威学习事实
-> 当前 Conversation
-> 当前 Notebook 来源与 Memory
-> 获得授权的 Personal Memory
-> 工具与输出预算
```

当记忆冲突、来源不可用或置信不足时，系统应明确不使用并询问，而不是补全一个看似连贯的
人物画像。

## 九、PostgreSQL 数据方案候选

第一阶段继续以 PostgreSQL 为事实源，候选表按职责拆分：

- `memory_items`：原子记忆、作用域、owner、类型、治理维度、版本和有效期；
- `memory_audiences`：成员、角色或私有主体的可见性关系及授权版本；
- `memory_sources`：消息、来源片段、权威事实或人工操作的可追溯关系；
- `memory_edges`：supersede、supports、contradicts、derived_from 等受限关系；
- `memory_candidates`：模型抽取、反思和确认状态；
- `memory_retrieval_logs`：Context Snapshot 中的选择、拒绝和预算证据；
- `memory_compactions`：Conversation 与 Notebook 摘要的输入版本和输出版本；
- `memory_suppressions`：以作用域、有效期、source/message ID、类别策略或 keyed fingerprint
  阻止被遗忘内容再次抽取；禁止保存原文、embedding 或可字典反推的裸 hash；
- `agent_memory_blocks`：可选的人工可编辑 Personal Memory 分区。

先使用结构化索引和 PostgreSQL 全文检索建立基线。只有评测证明语义召回明显不足后，才为
`pgvector` 建 ADR；只有关联推理证明有收益后，才启用 `memory_edges` 的复杂图检索。当前
没有引入 Redis、独立向量数据库或图数据库的依据。

## 十、更正、撤销与删除闭包

- 更正创建新版本，并将旧版本标记 `superseded`，保留不含敏感原文的审计关系；
- 撤销立即从 Context Compiler 排除，不等待后台物理删除；
- “忘记记忆但保留聊天”创建 suppression tombstone，阻止抽取任务再次生成同一语义候选；
- 删除 manifest 枚举原始记录、FTS/向量索引、edges、candidates、compactions、Snapshot 与检索
  日志、prompt/Trace、后台任务 payload、缓存、导出和对象存储派生物；
- 来源删除立即使派生摘要、候选和未来 compaction 失效；
- 物理清理通过可重试任务按存储介质 SLA 执行，审计只保留不可逆、低敏的操作元数据；
- 备份和外部模型 Provider 的保留限制必须在产品中诚实披露，并通过恢复后重放删除 manifest
  防止数据复活；
- 用户可以按 Personal、Notebook 和 Conversation 查看“Agent 记住了什么”；
- 每条可见记忆提供来源、作用域、更新时间、更正和删除入口；
- Guardian 是否能查看未成年人 Personal Memory 必须另立产品与隐私决策，不能默认继承。

## 十一、K12 隐私红线

禁止从聊天自动建立以下跨 Notebook 记忆：

- 医疗、心理诊断、性、宗教、政治倾向；
- 精确住址、实时位置、学校证件和生物特征；
- “聪明/懒惰/内向/多动”等人格或能力标签；
- 未经本人确认的年龄、性别、家庭关系和经济情况；
- 教师或家长对学生的私密评价。

系统可以根据当前任务临时调整表达，但不能将即时行为固化为人格。年龄和年级必须继续使用
显式声明及其来源，性别不是教学适配的默认输入。任何个性化都要能回答“为什么这样教我”。

M0 必须先接受年龄/身份/租户授权矩阵，至少覆盖：

- 未成年各年龄段、成年、年龄未知和匿名主体；
- 个人、Guardian 监督、学校/机构账号的数据控制者；
- 同意、撤回、监护关系变化、未成年转成年和账号合并；
- Personal Memory 的启用、查看、导出、删除与模型提取资格。
- Notebook 成员增删、角色变化，以及教师、本人、Guardian、同伴对个体记录的可见性。

矩阵未决定的组合 fail-closed，不进入 M1。

## 十二、评测设计

在 LongMemEval 五类能力之外增加 EduCanvas 专用夹具：

| 评测组     | 必须证明的行为                                        |
| ---------- | ----------------------------------------------------- |
| 作用域隔离 | 私人 Conversation 和成员证据不泄漏；三层无隐式提升    |
| 时间与更新 | 新版本胜出，旧事实不会被摘要重新激活                  |
| 更正与删除 | 立即停止使用，完整清除，保留聊天时不会重新抽取        |
| 来源治理   | 删除/撤权后派生记忆失效；共享成员不能投毒其他作用域   |
| 指令注入   | 恶意教材、摘要指令和 source-grounded 错误不获得指令权 |
| 权威冲突   | 判分、画像来源和权限始终胜过模型候选                  |
| 敏感信息   | 禁止项不被自动提取，不进入日志或 Trace                |
| 授权矩阵   | 年龄、匿名、监督、学校、成年转换均 fail-closed        |
| 拒答       | 缺少证据时询问，不编造过去偏好或经历                  |
| 检索质量   | 与无 Memory、全文检索、语义检索基线对照               |
| 成本与延迟 | 分层预算不会让普通回合的 token 和 P95 延迟失控        |
| 可解释性   | 每次调用可定位记忆 ID、版本、来源和选择原因           |

核心质量目标应优先降低错误记忆率和跨作用域泄漏率，而不是追求“每次都能找到一条记忆”。

## 十三、PageLM 能力映射

本节源码判断固定在 PageLM 提交
[`736f22b9b1b194fc50d90b29337d04d99ba81172`](https://github.com/CaviraOSS/PageLM/tree/736f22b9b1b194fc50d90b29337d04d99ba81172)。
[PageLM](https://github.com/CaviraOSS/PageLM) 值得学习的是学习产品表面的分工：Chat、
SmartNotes、Flashcards、Quiz、Podcast、Planner、Exam、Debate 和 Companion 被组织为专门
入口；后端也按能力路由拆分。EduCanvas 可将这些产品形态映射成 Notebook 内的
Artifact/Studio 工具架：

```text
Goal -> Sources -> Diagnostic -> Explain/Practice -> Feedback -> Progress
                        |
                        +-> Notes / Cards / Quiz / Audio / Experiment
```

但 PageLM 当前的轻量本地存储、全局 Keyv/SQLite 列表、文件读取型 Companion、宽松静态目录
与 AI 生成内容混合，不具备 EduCanvas 需要的用户/Notebook 作用域、可信事实和未成年人边界。
它是产品形态和模块拆分的学习对象，不是 Memory 数据架构模板。

## 十四、分阶段落地

### M0：契约、隐私与评测

- 接受三层作用域、正交治理维度和权威事实优先级 ADR；
- 接受年龄/身份/租户授权矩阵、成员 audience 与 Conversation ACL/跨层提升规则；
- 建立投毒、跨作用域、更新、删除、敏感信息和拒答夹具；
- 固定 Context Snapshot 的最小可解释字段。

### M1：显式可见的记忆

- 只向授权矩阵允许的主体开放 Personal Memory；Notebook Memory 提供查看、显式创建、更正
  与撤销；
- Web 为 K12 主入口，TUI 提供同一数据的高级操作；
- 暂不允许模型自动激活跨 Notebook 记忆。

### M2：来源型 Notebook Memory

- 来源摘要、Artifact 索引、Notebook compaction；
- Context Compiler 分层预算与 Snapshot；
- 来源删除和权限变化的级联失效。

### M3：候选提取与巩固

- 对成年或明确授权场景开启 `model_candidate`；
- 用户确认、冲突合并、过期和程序性反思；
- 评估个性化收益、错误记忆率和操作负担。

### M4：检索增强

- 以结构化/全文检索为基线进行语义检索对照；
- 达到预设收益门槛后再决定 `pgvector`；
- 图关系、自动演化和框架型 Memory 必须分别通过 ADR。

## 十五、建议形成的决策

建议负责人确认：

1. 接受 Personal、Notebook、Conversation 三层作用域；
2. 接受权威事实永远优先，Memory 不能覆盖专用事实表；
3. K12 Personal Memory 在年龄/身份/租户矩阵接受前全部关闭；
4. 第一阶段仅做显式、可见、可纠正、具有删除闭包的 Memory；
5. PostgreSQL 结构化索引与全文检索先行，向量和图能力以评测触发；
6. PageLM 用于学习产品表面与能力拆分，不作为信任或存储架构模板。

这些选择接受后，再创建 ADR、数据契约和 M0/M1 实现计划；在此之前不新增 Memory 表或运行时
依赖。
