# 唯一写者与故障恢复矩阵

- 状态：`verified design boundary`
- 核验日期：2026-07-21
- 依据：当前仓库源码、数据库约束、Provider/Tool/Continuation/Privacy/Trace fixtures

## 一、结论

Operation Event、Turn Ledger、Workflow Checkpoint 与 Worker Job 不是四份相同状态。它们分别回答“用户能观察到什么”“一次 Turn 实际发生了什么”“执行器下一步从哪里继续”“哪项工作等待哪个 worker”。第二代架构必须让每类事实只有一个写者，并禁止低层记录反向宣布高层终态。

```text
Gateway Operation Event（对外控制事实）
  -> Turn Ledger（消息、模型、工具与 Context 的业务执行事实）
    -> Workflow Checkpoint（可丢弃的执行游标）
      -> Worker Job（至少一次的调度租约）
```

箭头表示上层可以引用下层，不表示下层拥有上层状态。

## 二、唯一写者矩阵

| 记录                | 唯一语义写者                                                                   | 当前生产落点                                                                                        | 可表达                                                                | 禁止表达                                                                  | 恢复职责                                                                          |
| ------------------- | ------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| Operation Event     | Gateway Operation Application，经 `GatewayOperationStorePort.append`           | `DrizzleGatewayOperationStore` 同事务追加 `gateway_operation_events` 并更新 `agent_operations` 终态 | accepted、delta、approval、completed/failed/cancelled 与严格 sequence | 模型/worker/checkpoint 自行宣布 Operation 终态                            | 按 sequence replay 已提交事件；running 计算是否续跑由 continuation 决定           |
| Turn Ledger         | 统一 `TurnApplicationService` 经稳定 Ledger Port；迁移期现有仓储各写自己的旧表 | Teaching 为 `DrizzleTeachingTurnLedger`；General/Gateway 仍是待收敛双轨                             | 用户/老师消息、Model Run、Tool Call、Context Snapshot、每项内部终态   | 直接写 Gateway 终态；把 Trace 当审计；写可信学习结论                      | 以 client message/idempotency key replay；pending/lease 由原 writer 接管或回收    |
| Workflow Checkpoint | `OperationContinuationPort` 后唯一选中的 executor Adapter                      | 当前只有 Artifact checkpoint；LangGraph 仅研究 Saver                                                | 版本化控制状态、next step、等待原因、恢复游标                         | 身份、Notebook Membership、approval 事实、effect 成功、Operation/学习终态 | 从最后兼容 checkpoint 重启；恢复前重验主体/授权；不兼容时停机迁移或从业务账本重建 |
| Worker Job          | 业务事务中的 Job Scheduler/Outbox 创建；graphile-worker 只维护其调度租约       | `graphile_worker.add_job` 与各领域 job row 同事务提交                                               | task alias、最小业务 ID、attempt、runAt、lock/retry                   | Prompt/正文/Secret；以队列 complete 代替业务 complete                     | 至少一次重投；同一 effect key 对账；异常锁超时必须告警与人工/自动解锁策略         |

“唯一写者”是语义所有权，不要求所有写入只有一个进程。多个 Gateway 实例可以竞争同一 Operation advisory lock；多个 worker 可以抢任务，但都只能通过同一 Port 和数据库不变量改变该类事实。

## 三、当前双轨与迁移规则

当前还没有统一 Turn Ledger：Teaching 使用 `lesson_sessions/chat_messages/model_runs/tool_calls/turn_context_snapshots`，General 与 Gateway 使用 Conversation/Message/Operation 的另一条路径。它是第二代必须逐条迁移的事实，不应通过新增第三套表或框架 Session 掩盖。

迁移遵循以下规则：

1. 先定义统一 Ledger Port 与兼容读模型，再迁移一个入口；
2. 每个入口切换时只有一个写路径，禁止长期 dual-write；
3. 旧 ID 保留映射，学习事件继续能引用原 Model/Tool 证据；
4. 回滚切回旧 Adapter，不回滚已提交的业务事实；
5. 全部入口迁移并经过保留期后，才单独决策旧表清理。

## 四、故障点责任

| 故障点                         | 允许恢复者                    | 正确动作                                           | 错误动作                             |
| ------------------------------ | ----------------------------- | -------------------------------------------------- | ------------------------------------ |
| Event 已提交、客户端断线       | Gateway projection            | 从下一 sequence replay                             | 重新执行 Turn                        |
| 消息/Run 已开始、进程退出      | Turn Ledger lease owner       | 领取或回收 lease，读取既有 Ledger                  | 新建第二 Turn                        |
| effect 已提交、结果未落账      | Tool Kernel effect reconciler | 按 effect key 查询并标记 committed/outcome_unknown | worker 自动重复写                    |
| approval 已通过、runner 已丢失 | Continuation owner            | 原子领取 continuation，重新鉴权后续跑              | 仅因 checkpoint 表示 approved 就执行 |
| worker 异常退出                | graphile-worker + 运维策略    | 保留同一 job/effect key 重投并监控锁               | 创建新的业务 Operation 伪装恢复      |
| checkpoint 不兼容              | 版本迁移器/运维               | 停止领取、迁移或从 canonical ledger 重建           | 静默丢字段继续                       |

## 五、单终态不变量

- 对外唯一终态只由 Operation Event writer 提交；
- Turn 内 Model/Tool 的终态是证据，不自动成为 Operation 终态；
- checkpoint 的 `next=[]`、job 的 completed 或 Trace span end 都不是业务完成；
- 学习事实仍由 Teaching Domain Services 写入，不能由任一运行账本推断；
- effect ledger 的 committed 必须来自可验证副作用或同事务提交，超时写操作保持 `outcome_unknown`。

该矩阵是第二代迁移、回滚和故障注入的验收基线。任何框架若要求自己成为第二 Operation、Session/Notebook 或学习事实源，直接淘汰。
